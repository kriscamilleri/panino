# GitHub Backup — Spec

> Back up notes, folders, and images to a GitHub repository via force-push.
> Status: Draft — 2026-03-21

---

## 1) Summary

Provide a client-driven backup mechanism that authenticates with GitHub, lets the user select (or create) a repository, and force-pushes all notes, folder structure, and images to the `main` branch. This gives users a versioned, off-platform backup of their entire Panino workspace stored in their own GitHub account.

---

## 2) Goals

1. Authenticate with GitHub via OAuth (Device Flow or standard OAuth App).
2. Let users select an existing repository or create a new private one.
3. Export all notes as `.md` files, preserving the folder hierarchy.
4. Export all uploaded images into an `_assets/` directory.
5. Force-push the full workspace snapshot to the repo's `main` branch as a single atomic commit.
6. Run backups **automatically once per day** in the background, plus manual trigger from Tools.

## Non-Goals

- No pull/restore from GitHub in v1 (one-way push only).
- No incremental/diff-based pushes — each backup is a full snapshot.
- No branch selection — always targets `main`.
- No merge conflict handling — force-push overwrites history.
- No GitLab, Bitbucket, or other providers in v1.
- No user-configurable schedule — always daily.

---

## 3) UX & UI

### Entry Point

- **Tools → GitHub Backup**.

### Connection Flow

1. User clicks "Connect GitHub Account".
2. App initiates OAuth flow (GitHub OAuth App or Device Flow for desktop-friendliness).
3. On success, display connected GitHub username and avatar.
4. "Disconnect" button to revoke and clear tokens.

### Repository Selection

- After connecting, show a dropdown of the user's repositories (filtered to repos with push access).
- "Create New Repository" option that prompts for a repo name (default: `panino-backup`) and creates it as **private**.
- Selected repo is saved and persisted for future backups.

### Backup Actions

- **"Back Up Now"** button — triggers an immediate manual backup.
- Progress indicator showing: exporting notes → building tree → pushing to GitHub.
- Last backup timestamp displayed.
- **Daily auto-backup**: Runs automatically once per day in the background when the user is logged in. No user configuration needed — enabled by default once a GitHub account and repo are connected.

### Backup Status

- Show last backup date/time (auto or manual), repo name, and commit SHA.
- Next scheduled backup time.
- Error state if last backup failed, with retry option and error message.

## 4) Backup Format

The repository will contain a clean, human-readable file tree:

```
panino-backup/
├── README.md                   # Auto-generated index
├── _assets/
│   ├── image-uuid-1.png
│   ├── image-uuid-2.jpg
│   └── ...
├── My Folder/
│   ├── Note One.md
│   └── Subfolder/
│       └── Note Two.md
├── Another Folder/
│   └── Note Three.md
└── Unfiled Note.md
```

### File Naming

- Note filenames derived from note title (sanitized for filesystem: replace `/\:*?"<>|` with `-`, truncate to 200 chars).
- Duplicate titles in the same folder get a numeric suffix: `Note (2).md`.
- Images retain their stored filenames (UUID-based).

### README.md

Auto-generated file at the repo root:

```markdown
# Panino Backup

Backed up on: 2026-03-21T14:30:00Z
Notes: 142 | Folders: 23 | Images: 57

This repository is auto-generated by [Panino](https://github.com/kriscamilleri/panino).
Do not edit files here — they will be overwritten on the next backup.
```

## 5) Implementation

### Authentication

- Use GitHub OAuth App flow (backend-mediated) or GitHub Device Flow (frontend-only).
- Store GitHub access token encrypted in the backend per-user database.
- Required scopes: `repo` (full access to private repos).

### Git Operations

Performed on the **backend** to avoid shipping a full Git client to the browser:

1. Backend receives "backup now" request from authenticated user.
2. Read all notes, folders, and images from the user's database/uploads.
3. Build an in-memory Git tree using a library like `isomorphic-git` or GitHub's Git Data API (Trees + Blobs + Commits).
4. Create a commit with the full snapshot.
5. Force-push to `refs/heads/main`.

**Preferred approach**: Use the **GitHub REST API** (Git Data endpoints) to avoid needing Git installed on the server. The entire workspace is committed as a **single atomic operation** — one tree, one commit, one ref update:

1. `POST /repos/:owner/:repo/git/trees` — create a single tree containing all files inline (blobs specified with `content` field rather than separate blob creation). GitHub supports trees up to 100k entries.
2. `POST /repos/:owner/:repo/git/commits` — create one commit pointing to the tree.
3. `PATCH /repos/:owner/:repo/git/refs/heads/main` — force-update the ref.

This keeps the backup to **~3 API requests** regardless of workspace size, well within rate limits. For files exceeding the inline content limit (>100 KB), fall back to creating individual blobs via `POST /repos/:owner/:repo/git/blobs` first.

### API Endpoints

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/backup/github/connect` | Initiate GitHub OAuth flow |
| `GET` | `/backup/github/callback` | OAuth callback |
| `DELETE` | `/backup/github/disconnect` | Revoke and clear GitHub token |
| `GET` | `/backup/github/repos` | List user's repositories |
| `POST` | `/backup/github/repos` | Create a new private repository |
| `POST` | `/backup/github/run` | Execute a full backup now |
| `GET` | `/backup/github/status` | Get last backup status/timestamp |

### Data Model

#### Table: `backup_config` (backend, per-user DB, not CRR)

| Column | Type | Notes |
|--------|------|-------|
| `id` | TEXT PK | UUID |
| `provider` | TEXT | `"github"` |
| `access_token_enc` | TEXT | Encrypted GitHub token |
| `username` | TEXT | GitHub username |
| `repo_full_name` | TEXT | `owner/repo` |
| `auto_backup_enabled` | INTEGER | 1 = daily auto-backup (default once connected) |
| `last_backup_at` | TEXT | ISO 8601 |
| `last_backup_sha` | TEXT | Commit SHA |
| `last_error` | TEXT | Null if successful |
| `created_at` | TEXT | ISO 8601 |

## 6) Performance & Limits

- The single-tree approach keeps each backup to **~3 API requests** (tree + commit + ref update), plus individual blob uploads only for files >100 KB. This is well within GitHub's 5,000 requests/hour rate limit.
- GitHub API has a blob size limit of **100 MB per file** — unlikely to hit for notes but relevant for large images.
- Daily auto-backup runs as a background job on the backend. The backend tracks last backup time per user and triggers a new backup if >24 hours have elapsed when the user syncs.
- Manual backups via "Back Up Now" run immediately; user can navigate away. Status polled via `/backup/github/status`.

---

## 7) Security Considerations

- GitHub token encrypted at rest (same scheme as calendar tokens).
- Token never sent to the frontend — all GitHub API calls happen server-side.
- Created repos default to **private**.
- Sanitize note titles before using as filenames (prevent path traversal).
- Rate-limit the backup endpoint to prevent abuse (max 1 concurrent backup per user).
